# 🎨 UI Architecture

Документ описывает архитектуру UI-слоёв проекта и правила их взаимодействия.

Архитектура построена по принципу:

> **Один слой — одна ответственность**
> UI не работает напрямую с API и не содержит бизнес-логики.

---

## 🔄 Общая схема UI-потока

```text
CONFIG
  ↓
NORMALIZER (опционально)
  ↓
ADAPTER
  ↓
VIEW MODEL (опционально)
  ↓
RENDERER
  ↓
BLOCK (UI)
```

---

## 1️⃣ CONFIG — конфигурация интерфейса

### Назначение

Описывает структуру UI и управляет тем:

-   что отображается
-   в каком порядке
-   какие слои участвуют

**CONFIG** — это точка входа UI и основной управляющий слой.

### Отвечает за

-   набор блоков на странице
-   порядок блоков
-   параметры отображения
-   связи:

    -   `block → adapter`
    -   `adapter → viewModel`

-   включение / отключение функциональности
-   вариативность UI без изменения компонентов

### Где находится

```text
src/lib/config/
```

### Примеры файлов

-   `app.config.js`
-   `blocks.config.js`
-   `card.config.js`
-   `list.config.js`
-   `detail.config.js`

### Принципы

-   ❌ не содержит логики
-   ❌ не использует stores
-   ❌ не знает про DOM
-   ✅ управляет всей UI-архитектурой
-   ✅ является источником истины для UI

---

## 2️⃣ NORMALIZER — нормализация данных (опционально)

### Назначение

Приводит входные данные к форме, ожидаемой адаптерами и UI.

Используется, когда:

-   данные приходят из разных источников
-   структура данных нестабильна
-   нужно выбрать переведённые поля
-   нужно выровнять формат сущностей

### Отвечает за

-   выбор локализованных полей
-   выравнивание структуры данных
-   удаление лишних серверных полей
-   приведение данных к единому формату

### Где находится

```text
src/lib/utils/
src/lib/normalizers/
```

### Принципы

-   ❌ не содержит бизнес-логики
-   ❌ не форматирует данные для UI
-   ❌ не использует stores
-   ✅ подготавливает данные для адаптера

---

## 3️⃣ ADAPTER — бизнес-логика блока

### Назначение

Преобразует данные сущности (`item`) в готовый набор данных для конкретного UI-блока.

### Сигнатура

```js
Adapter({ item, system }) → PlainObject
```

### Отвечает за

-   вычисление бизнес-состояний
-   работу со скидками, статусами, флагами
-   подготовку данных под конкретный блок
-   сбор сложных значений (таймеры, лейблы, условия)

### Где находится

```text
src/lib/adapters/
src/lib/features/*/*.adapter.js
```

### Принципы

-   ❌ не реактивен
-   ❌ не использует stores
-   ❌ не знает про UI
-   ❌ не форматирует валюты и даты
-   ✅ возвращает плоский объект данных

---

## 4️⃣ VIEW MODEL (VM) — реактивный слой (опционально)

### Назначение

Добавляет реактивность поверх данных, подготовленных адаптером.

Используется только там, где это действительно нужно.

### Отвечает за

-   реакцию на смену валюты
-   таймеры
-   live-обновления
-   пользовательские настройки
-   работу со Svelte stores

### Где находится

```text
src/lib/features/*/*.vm.js
```

### Принципы

-   ✅ основан на данных адаптера
-   ✅ использует stores
-   ✅ может быть derived
-   ❌ не содержит бизнес-логики
-   ❌ не знает про структуру UI

Если данные загружаются через server load
и после изменений перечитываются через invalidate —
VIEW MODEL для них не используется.

---

## 5️⃣ RENDERER — связывающий слой

### Назначение

Соединяет:

-   CONFIG
-   ADAPTER
-   VIEW MODEL
-   UI-компонент

### Отвечает за

-   вызов адаптера
-   инициализацию VM
-   передачу данных в блок
-   выбор нужного UI-компонента

### Где находится

```text
src/lib/components/renderers/
```

### Принципы

-   ❌ не содержит бизнес-логики
-   ❌ не содержит разметки
-   ✅ только связывает слои

---

## 6️⃣ BLOCK (UI) — визуальный компонент

### Назначение

Отображает данные, полученные от renderer.

### Отвечает за

-   HTML-разметку
-   стили
-   минимальные UI-условия (`{#if}`, `{#each}`)

### Где находится

```text
src/lib/components/blocks/
```

### Принципы

-   ❌ не знает про API
-   ❌ не знает про бизнес-логику
-   ❌ не подписывается на stores напрямую
-   ✅ максимально тупой и предсказуемый

---

## 🧠 Ключевые правила архитектуры UI

-   CONFIG управляет всем UI
-   ADAPTER — единственное место бизнес-логики
-   VM используется только при необходимости реактивности
-   UI-компоненты не думают
-   Реактивность — на границе, а не внутри блоков
-   Каждый слой можно заменить без переписывания остальных

---

## ✅ Итог

Такая архитектура позволяет:

-   масштабировать UI без хаоса
-   добавлять новые типы блоков через конфиг
-   переиспользовать бизнес-логику
-   держать UI чистым и управляемым
